## Mandatory Lookup Workflow

When a task involves ESP32-S3 hardware facts, do not rely on memory alone. First inspect the relevant processed manual artifacts.

Recommended order:

1. Read `<processed-root>/manifest.json` to check status, page count, source PDF, table count, and alert count.
2. If `alert_count > 0`, read `<processed-root>/alerts.json` before trusting table-heavy or figure-heavy content.
3. Search `<processed-root>/sections.jsonl` for chapter/topic navigation.
4. Search `<processed-root>/chunks.jsonl` for page-aware retrieved facts and citations.
5. Read `<processed-root>/document.md` around the relevant heading for context.
6. Use `<processed-root>/tables/*.csv` or `<processed-root>/tables/*.html` for table values.
7. Use `<processed-root>/document.html`, `<processed-root>/artifacts/`, and the original PDF for diagrams, pin matrices, register bit diagrams, timing figures, and anything flagged in `alerts.json`.
8. For register values, bit definitions, pin mappings, timing limits, and electrical characteristics, cite the manual page and cross-check against the original PDF when the answer affects code or hardware behavior.

## Known Manual Processing Caveats

Do not treat processed Markdown as the final authority. Known Docling boundaries for current ESP32-S3 artifacts include:

- The datasheet has an alert for `Table 2-9. Peripheral Pin Assignment`, where the table body fell back to an image.
- The TRM has `empty_table_sidecar` alerts for several empty or unusable table sidecars.
- Very large TRM sections may contain formula serialization markers such as `formula-not-decoded`.
- Markdown may contain page furniture and broken words such as `T able`.

When a processed artifact looks suspicious, inspect `alerts.json`, the relevant `artifacts/` image, `document.html`, and the original PDF page.

## Editing ESP-IDF Code

For ESP-IDF projects in this repository:

- Prefer project-local ESP-IDF conventions and existing `CMakeLists.txt` / `sdkconfig` structure.
- Do not change generated build directories unless explicitly asked.
- Do not rely on register-level assumptions without checking the ESP32-S3 TRM processed artifacts.
- If using GPIO, I2C, SPI, UART, interrupts, clocks, power, boot straps, or pin functions, check the datasheet/TRM first and include page references in the final explanation when relevant.
